# aide-dev-test: テスト実行・証跡収集・レポート生成

テスト仕様書（単体/結合）に基づいてテストを実行し、証跡を自動収集し、テストレポートを生成する。

## テストケースID体系

| 種別 | ID形式 | 例 |
|---|---|---|
| 単体テスト | `TC-UT-{機能略称}-{連番}` | TC-UT-AUTH-001 |
| 結合テスト | `TC-IT-{機能略称}-{連番}` | TC-IT-AUTH-001 |

証跡ファイル名はテストケースIDと完全一致させる。

## 手順

### 1. 事前読み込み

1. 対象機能のテスト仕様書を読む:
   - `docs/04_testing-テスト/artifacts-成果物/単体テスト仕様書-{機能名}.md`
   - `docs/04_testing-テスト/artifacts-成果物/結合テスト仕様書-{機能名}.md`
2. 対象機能の設計書（`docs/02_design/artifacts-成果物/設計書-{機能名}.md`）を読む
3. 対象機能のソースコード（`src/` 配下）とテストコード（`tests/` 配下）を読む
4. `docs/00_pm-管理/技術スタック選定.md` からテストツール設定を確認

他の履歴ファイルは読まない。

### 2. テスト種別の確認

ユーザーに以下を確認する（引数で指定されている場合はスキップ）：

- **テスト種別**: `unit`（単体） / `integration`（結合） / `both`（両方）
- **対象機能**: 機能名
- **実行モード**: `full`（全テストケース） / `failed`（前回失敗分のみ）

### 3. テスト実行と証跡収集

テストケースごとに以下を実行する：

#### 3-A. CLI/APIテストの証跡収集

```bash
# テスト実行してログをキャプチャ
{テストコマンド} 2>&1 | tee /tmp/test-output-{TC-ID}.log

# ansi2html でHTML化
cat /tmp/test-output-{TC-ID}.log | ansi2html > evidence/{種別}/{機能名}/{TC-ID}.html
```

- テスト実行コマンドの stdout/stderr をすべてキャプチャ
- `ansi2html` でカラーコード付きHTMLに変換
- 証跡ファイル名はテストケースIDと一致させる

#### 3-B. 画面テストの証跡収集（Playwright）

```javascript
// Playwright でスクリーンショットを自動取得
// テストケースの手順に対応するステップごとにキャプチャ
await page.screenshot({
  path: `evidence/{種別}/{機能名}/{TC-ID}-{ステップ連番}.png`,
  fullPage: true
});
```

- テスト仕様書の「手順」に記載された各ステップでスクリーンショットを取得
- ファイル名: `{TC-ID}-{ステップ連番}.png`（例: `TC-UT-AUTH-001-01.png`）
- 実行ログも併せて `{TC-ID}.html` として保存

### 4. 証跡ファイルの配置

```
docs/04_testing-テスト/evidence-証跡/
├── unit-単体/
│   └── {機能名}/
│       ├── TC-UT-AUTH-001.html          ← コマンドログ
│       ├── TC-UT-AUTH-001-01.png        ← スクリーンショット（画面テスト時）
│       ├── TC-UT-AUTH-001-02.png
│       └── TC-UT-AUTH-002.html
└── integration-結合/
    └── {機能名}/
        ├── TC-IT-AUTH-001.html
        ├── TC-IT-AUTH-001-01.png
        └── TC-IT-AUTH-002.html
```

### 5. 証跡ドキュメントの生成

テスト仕様書とリンクした証跡一覧ドキュメントを生成する：

出力先:
- `docs/04_testing-テスト/artifacts-成果物/単体テスト証跡-{機能名}.md`
- `docs/04_testing-テスト/artifacts-成果物/結合テスト証跡-{機能名}.md`

```markdown
# 単体テスト証跡: {機能名}

## 基本情報
- 対象機能: {機能名}
- テスト仕様書: 単体テスト仕様書-{機能名}.md
- 実行日: YYYY-MM-DD
- 実行環境: {OS, ランタイムバージョン等}
- テストツール: {Jest / pytest 等}

## 証跡一覧

| テストケースID | テストケース名 | 結果 | 証跡ファイル | スクリーンショット |
|---|---|---|---|---|
| TC-UT-AUTH-001 | 正常ログイン | ✅ PASS | [ログ](../evidence-証跡/unit-単体/認証/TC-UT-AUTH-001.html) | [SS1](../evidence-証跡/unit-単体/認証/TC-UT-AUTH-001-01.png), [SS2](../evidence-証跡/unit-単体/認証/TC-UT-AUTH-001-02.png) |
| TC-UT-AUTH-002 | パスワード不正 | ❌ FAIL | [ログ](../evidence-証跡/unit-単体/認証/TC-UT-AUTH-002.html) | — |

## 失敗テストケースの詳細

### TC-UT-AUTH-002: パスワード不正
- **期待結果**: エラーメッセージ「パスワードが正しくありません」を表示
- **実際の結果**: 500 Internal Server Error
- **原因分析**: [調査結果]
- **対応**: issues-課題.md に起票 → I-XXX
```

### 6. テストレポートの生成

テスト結果を集約したサマリレポートを生成する：

出力先:
- `docs/04_testing-テスト/artifacts-成果物/単体テストレポート.md`
- `docs/04_testing-テスト/artifacts-成果物/結合テストレポート.md`

```markdown
# 単体テストレポート

## サマリ

| 項目 | 値 |
|---|---|
| 実行日 | YYYY-MM-DD |
| 対象機能数 | X |
| テストケース総数 | XX |
| 合格 | XX (XX%) |
| 不合格 | XX (XX%) |
| 未実行 | XX (XX%) |
| 合格基準 | 全テストケース合格 |
| 判定 | ✅ 合格 / ❌ 不合格 |

## 機能別結果

| 機能 | テスト数 | 合格 | 不合格 | 未実行 | 合格率 |
|---|---|---|---|---|---|
| ユーザー認証 | 15 | 14 | 1 | 0 | 93.3% |
| タスク管理 | 20 | 20 | 0 | 0 | 100% |
| チーム管理 | 12 | 12 | 0 | 0 | 100% |

## 不合格テストケース一覧

| テストケースID | 機能 | テストケース名 | 原因 | 課題ID | 対応状況 |
|---|---|---|---|---|---|
| TC-UT-AUTH-002 | ユーザー認証 | パスワード不正 | 実装不備 | I-XXX | 修正中 |

## 品質判定

### 合格基準
- [ ] 全テストケースが合格（必須）
- [ ] 重大度「高」の不合格が0件（必須）
- [ ] 合格率95%以上（推奨）

### 判定結果
[合格基準の達成状況と総合判定]
```

### 7. 不合格時の課題起票

テストが不合格の場合：
1. `docs/00_pm-管理/issues-課題.md` に課題として起票
2. 仕様の不備か実装の不備かを切り分けて記録
3. ユーザーに報告し、対応方針を確認

### 8. deliverables-成果物一覧.md の更新

`docs/04_testing-テスト/deliverables-成果物一覧.md` の対応する成果物の状態を更新する。

## 前提条件

以下のツールが必要（技術スタック選定.md で決定済みであること）：

| ツール | 用途 | 必須/任意 |
|---|---|---|
| テストフレームワーク | テスト実行（Jest / pytest 等） | 必須 |
| ansi2html | コマンドログのHTML化 | 必須 |
| Playwright | 画面テストのスクリーンショット取得 | 画面テスト時必須 |

未インストールの場合はインストール手順を案内する。

## テスト完了後の案内

テストが完了したら、以下の次ステップを案内する：

> 品質レポートを生成しますか？ → `/aide-review-code --scope coverage,security`
> 全観点でレビューする場合: `/aide-review-code`

## 注意事項

- テスト仕様書のテストケースIDと証跡ファイル名は必ず一致させる
- 証跡は上書きせず、実行日ごとにフォルダ分けするか、Gitで履歴管理する
- テスト結果は改ざんしない（失敗は正直に記録する）
- 大量のスクリーンショットが生成される場合、evidence/ フォルダは .gitignore に追加を推奨
