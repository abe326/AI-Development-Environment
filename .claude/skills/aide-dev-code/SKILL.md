# aide-dev-code: 仕様駆動コード作成

仕様を正としてコードを作成・修正する。実装前に必ずユーザーの承認を取る。

## 手順

### 1. 事前読み込み（コンテキスト最小読み込み）

1. docs/00_pm-管理/issues-課題.md を読み、対象の課題・タスクを特定する
2. docs/02_design/artifacts-成果物/設計書.md を読む（設計の全体像）
3. **対象機能の設計書（docs/02_design/artifacts-成果物/設計書-{機能名}.md）を読む** ← 実装の正となる仕様
4. **対象機能の製造計画書（docs/03_plans/artifacts-成果物/製造計画書-{機能名}.md）を読む** ← AIへの実装指示書（実装方針・処理フロー・入出力定義・テストケース）
5. 修正の場合: 対象の既存ソースコードを読む

他の履歴ファイルは読まない。

### 2. 実装前の確認（必須 — 承認なしで実装に進まない）

製造計画書の各セクションを活用して、以下の内容をユーザーに提示し、承認を得る：

- **準拠する仕様**: どの設計書・製造計画書に基づくか（← 製造計画書 §1 基本情報）
- **対象ファイル**: 何を作成/修正するか（← 製造計画書 §2 ファイル構成）
- **処理フロー**: 正常系・異常系の概要（← 製造計画書 §3 処理フロー）
- **入出力**: API/関数のシグネチャ（← 製造計画書 §4 入出力定義）
- **依存関係**: 他機能との接続点（← 製造計画書 §5 依存関係）
- **実装方針**: アプローチの概要（← 製造計画書 §6 実装方針）
- **仕様に記載がない点**: 仕様書に記載がなく、判断が必要な点（あれば仕様の補完を提案）

**ユーザーが「OK」「進めて」等の承認を出すまで実装に着手しないこと。**

### 3. サブエージェントによる並列実装

承認後、以下を並列で実行する：

**Sub-A: 仕様に基づく実装コード生成**
- 設計書・製造計画書に記載された要件・振る舞いに準拠してコードを作成
- 製造計画書の処理フローに従って実装
- 段階的生成: スケルトン → ロジック → エッジケース

**Sub-B: 仕様準拠チェック**
- 生成コードと設計書・製造計画書を照合
- 仕様との差異があれば報告

**Sub-C: 既存テストの実行（あれば）**
- テストを実行し、パスすることを確認

### 4. 結果の統合

- 仕様との差異があれば報告
- コミットメッセージに関連する仕様（設計書・製造計画書のファイル名等）を記載

### 5. コーディング規約

- CLAUDE.md に記載されたコーディング規約に従う
- 1PR = 1タスクを原則とする
- コミットメッセージは Conventional Commits に従う

## 仕様に不備がある場合

仕様が曖昧・不足・矛盾している場合は、**実装を進めずにユーザーに報告**すること。
必要に応じて /aide-pm-brainstorm で検討し、仕様を補完してから実装する。

## 注意事項

- 仕様に記載がない機能を独自判断で追加しない
- セキュリティに関わるコードは特に慎重にレビュー
- `.env` や認証情報はコードに含めない
