# aide-pm-slide: 資料作成

4種類の資料を作成する。利用者の指示内容から適切な種類を自動判定し、テンプレートを使って生成する。

## 対応する資料の種類

| # | 種類 | 形式 | テンプレート | 用途 |
|---|---|---|---|---|
| 1 | **スライド** | Marp md → HTML | slide-template.md + marp-theme.css | プレゼン・説明・ヒアリング |
| 2 | **見積書** | HTML | estimate-template.html + document-style.css | 工数・金額の一覧表 |
| 3 | **スケジュール** | HTML | schedule-template.html + document-style.css | ガントチャート |
| 4 | **ディスカッション資料** | HTML | discussion-template.html + document-style.css | 課題説明・打合せ検討用 |

## 手順

### 1. 資料の種類の判定

ユーザーの指示から資料の種類を自動判定する。判定できない場合は質問形式で確認する。

#### 自動判定のルール

| ユーザーの指示に含まれるキーワード | 判定結果 |
|---|---|
| プレゼン、説明、報告、紹介、提案、ヒアリング、スライド | → スライド |
| 見積、工数、コスト、金額、予算 | → 見積書 |
| スケジュール、ガント、タイムライン、計画、日程、マイルストーン | → スケジュール |
| 検討、議論、ディスカッション、課題、論点、打合せ用、会議用 | → ディスカッション資料 |

#### 判定できない場合の質問

> どのような資料を作成しますか？
>
> 1. **スライド** — プレゼン・説明・ヒアリング用（Marp形式）
> 2. **見積書** — 工数・金額の一覧表
> 3. **スケジュール** — ガントチャート・マイルストーン
> 4. **ディスカッション資料** — 課題の論点整理・打合せ検討用

### 2. テンプレートの読み込み

種類に応じたテンプレートを読み込む：

| 種類 | 読み込むテンプレート |
|---|---|
| スライド | `.aide/templates/export/slide-template.md`, `.aide/templates/export/marp-theme.css`, `.aide/templates/export/metadata.yaml` |
| 見積書 | `.aide/templates/export/estimate-template.html`, `.aide/templates/export/document-style.css` |
| スケジュール | `.aide/templates/export/schedule-template.html`, `.aide/templates/export/document-style.css` |
| ディスカッション | `.aide/templates/export/discussion-template.html`, `.aide/templates/export/document-style.css` |

### 3. コンテキスト収集

種類に応じて必要な情報を収集する。

**全種類共通:**
- CLAUDE.md（プロジェクト概要）
- docs/00_pm-管理/issues-課題.md（OPEN課題）

**スライド:**
- ユーザーがテーマを指定 → その指示に従う
- 指定なし → 最新の成果物（要件書・設計書・製造計画書）に基づく

**見積書:**
- docs/00_pm-管理/estimate-見積もり.md（存在する場合）
- 各フェーズの deliverables-成果物一覧.md（成果物の量から工数を参照）
- 要件書・設計書（スコープの確認）

**スケジュール:**
- docs/00_pm-管理/schedule-スケジュール.md（存在する場合）
- issues-課題.md（残課題の影響）

**ディスカッション資料:**
- ユーザーが指定した論点・テーマ
- 関連するbrainstormsファイル
- issues-課題.md（関連する未決事項）

### 4. 対象読者の確認（スライドのみ）

スライドの場合、対象読者が不明であれば確認する：
- 技術者向け（詳細な技術情報を含む）
- 非技術者向け（ビジネス価値・概要中心）
- 経営層向け（コスト・スケジュール・リスク中心）

### 5. ストーリー構成の設計（スライドのみ）

スライドの全体構成は **SCQ（Situation → Complication → Question → Answer）** をベースに設計する。
各セクション内部は **ピラミッド原則（結論→根拠→詳細）** で整理し、情報分解は **MECE（漏れなくダブりなく）** を守る。

#### 学術的背景

| フレームワーク | 出自 | 役割 |
|---|---|---|
| SCQ（SituComBA） | Barbara Minto, McKinsey（1985） | スライド全体のストーリーライン制御 |
| ピラミッド原則 | Barbara Minto, "The Pyramid Principle" | 各セクション内の論理階層 |
| MECE | McKinsey（1960s） | 情報分解の網羅性・排他性 |
| 認知負荷理論 | John Sweller（1988） | 1スライドの情報量制御（7±2チャンク） |
| チャンキング | George Miller（1956） | 情報の塊分け |

#### SCQ構成の基本フロー

```
表紙 → サマリー(A要約) → 背景(S) → 課題(C) → 論点(Q) → 提案(A詳細) → 補足 → Next Action
```

| フェーズ | 役割 | スライド目安 | 使用パターン |
|---|---|---|---|
| **表紙** | タイトル・日付・発表者 | 1枚 | `<!-- _class: lead -->` |
| **サマリー** | 結論の先出し（Answer要約） | 1枚 | `<!-- _class: summary -->` |
| **S: Situation（背景）** | 前提・現状の共有 | 2-3枚 | 箇条書き、フロー、図解 |
| **C: Complication（課題）** | 問題提起・痛みの提示 | 2-3枚 | `<!-- _class: issue -->`、Before/After |
| **Q: Question（問い）** | 聞き手の疑問を代弁 | 0-1枚 | 強調ポイント（引用） |
| **A: Answer（提案）** | 解決策の詳細提示 | 3-5枚 | 比較、選定結果、推奨案 |
| **補足** | スケジュール・リスク・コスト | 1-3枚 | タイムライン、リスク一覧、KPI |
| **Next Action** | 次のアクション・決定事項 | 1枚 | `<!-- _class: cta -->` |

#### ピラミッド原則の適用（各セクション内部）

Answer（提案）セクションでは、主張→根拠の階層構造で整理する：

```
主張（スライドタイトル = 結論）
├── 根拠1（MECE: 観点A）
│   ├── データ・事実
│   └── 具体例
├── 根拠2（MECE: 観点B）
│   ├── データ・事実
│   └── 具体例
└── 根拠3（MECE: 観点C）
    ├── データ・事実
    └── 具体例
```

#### スライドタイトルのルール

h2（スライドタイトル）は**そのスライドの主張・結論**を書く。単なるラベル（「課題について」）ではなく、メッセージ（「認証方式の未決定が設計着手を阻害している」）にする。

#### 対象読者による構成の調整

| 対象 | S（背景）| C（課題）| A（提案）| 補足 |
|---|---|---|---|---|
| 技術者 | 軽め（既知前提） | 技術的課題を詳細に | 実装詳細・アーキテクチャ | 技術的リスク・制約 |
| 非技術者 | 丁寧に（業務視点） | 業務影響を強調 | ビジネス価値中心 | コスト・スケジュール |
| 経営層 | 最小限 | インパクト（金額・期間） | ROI・費用対効果 | リスクと対策のみ |

#### Questionの省略判定

Questionスライドは明示的に置かなくてもよい。ComplicationからAnswerへの流れが自然な場合は省略する。ただし、複数の選択肢がある提案型スライドでは「どの方式を採用すべきか？」のようにQuestionを明示するのが効果的。

### 6. フェーズの判定と出力先の決定

資料の内容からフェーズを判定し、出力先を決定する。

| 内容 | フェーズ | 出力先 |
|---|---|---|
| プロジェクト横断・見積・スケジュール | PM | `docs/00_pm-管理/slides-資料/` |
| 要件に関する資料 | 要件定義 | `docs/01_requirements/slides-資料/` |
| 設計に関する資料 | 設計 | `docs/02_design/slides-資料/` |
| 製造計画に関する資料 | 製造計画 | `docs/03_plans/slides-資料/` |

### 7. 資料の生成

#### スライドの場合

Marp形式mdを生成する。

ファイル名: `YYYYMMDD-テーマ.md`

フロントマター:
```yaml
---
marp: true
theme: aide-corporate
paginate: true
footer: "YYYY-MM-DD | [会社名/部署名]"
---
```

**重要: theme は必ず `aide-corporate` を指定する。header は使用しない。**

スライド構成はslide-template.mdの29パターンから適切なものを選んで使用する。

スライド作成ルール:
1. **1スライド1メッセージ**: 詰め込みすぎない
2. **h2 = スライドタイトル**: 固定位置に表示される
3. **箇条書き中心**: 長文は避ける
4. **図の活用**: .drawio.svg があれば参照
5. **Marpディレクティブ活用**: `<!-- _class: lead/section/summary -->` を使い分ける
6. **スライド枚数**: 10-20枚

#### コンテンツ品質ルール（内容の深さ）

**フォーマットが完璧でも、内容が薄ければ資料としての価値はない。** 以下のルールを守る。

##### 「薄い内容」の禁止

| NG例 | 問題 | OK例 |
|---|---|---|
| 「AIツールが普及している」 | 誰でも知っている一般論 | 「GitHub Copilot導入企業の77%がコード生成のみに利用」 |
| 「生産性が向上する」 | 定量性がない | 「ドキュメント作成時間を平均40%削減」 |
| 「品質が改善する」 | 具体性がない | 「レビュー観点の標準化で指摘漏れを半減」 |
| 「属人化が課題」 | 問題の構造が見えない | 「メンバー交代時に引き継ぎに2週間かかる」 |

##### 各スライドに必須の要素

| SCQフェーズ | 必須要素 |
|---|---|
| S（背景） | **具体的な数字・事実**。「現在〜である」を裏付けるデータ |
| C（課題） | **痛みの定量化**。コスト・時間・リスクを数字で示す |
| A（提案） | **具体的なメカニズム**。「なぜ解決できるのか」の仕組み |
| 補足 | **実績・比較データ**。導入前後の変化を具体的に |

##### コンテンツ深度チェック

スライド生成後、各スライドに対して以下を自己チェックする：

1. **So What?テスト**: そのスライドを見た聞き手が「だから何？」と思わないか
2. **数字テスト**: 少なくとも2枚に1枚は具体的な数値が含まれているか
3. **固有名詞テスト**: 一般論ではなく、固有の事実・ツール名・事例が含まれているか
4. **行動テスト**: 聞き手がそのスライドを見て何かを判断・行動できるか

##### コンテキストからの情報抽出

スライドの内容を充実させるため、以下のソースから具体情報を積極的に抽出する：

| ソース | 抽出する情報 |
|---|---|
| CLAUDE.md | プロジェクト固有の数値・制約・目標 |
| issues-課題.md | 具体的な課題名・影響・対応状況 |
| 要件書・設計書 | 機能一覧・非機能要件・技術選定理由 |
| brainstorms/ | 検討過程で出た比較データ・根拠 |
| estimate-見積もり.md | 工数・コスト・期間の具体数値 |

**情報が不足している場合は、薄い内容で埋めるのではなく、ユーザーに「この部分に入れるべき具体的なデータはありますか？」と質問する。**

#### ビジュアルデザインルール（CRAP原則 + タイポグラフィック・スケール）

スライドの視覚品質を保つため、以下のルールを**必ず**守る。

##### タイポグラフィック・スケール（文字サイズ階層）

marp-theme.css にCSS変数として定義済み。md生成時はこの階層を意識する。

| 階層 | サイズ | 用途 | Markdown記法 |
|---|---|---|---|
| h1 | 39px | 表紙タイトル | `# タイトル` |
| h2 | 31px | スライドタイトル（主張） | `## タイトル` |
| h3 | 25px | セクション内見出し・項目名 | `### 見出し` |
| 本文 | 20px | 箇条書き・説明文 | `- テキスト` |
| 補足 | 17px | 表セル・図形内テキスト | テーマCSSが自動適用 |
| 注釈 | 14px | 補足説明・注釈 | `<small>テキスト</small>` |
| キャプション | 12px | 出典・図の説明 | `<span class="caption">テキスト</span>` |

##### 余白率ルール（20%下限）

**余白率は20%を下回らないこと。同時に30%を超えるとスカスカに見えるため、コンテンツ系スライドでは20〜30%に収める。**

- 余白率20%未満 → コンテンツを削るかスライドを分割する
- 余白率30%超 → コンテンツが薄い。情報を追加するかレイアウトを見直す
- 表紙・セクション区切りは例外（余白が多くてよい）

スライドの種類ごとの目安：

| スライドの種類 | 余白率の目安 |
|---|---|
| 表紙・セクション区切り | 40〜60%（視覚的インパクト重視） |
| 説明系（箇条書き・テキスト） | 20〜30% |
| データ系（表・グラフ・図） | 20〜25% |
| 比較系（2カラム・Before/After） | 20〜30% |

余白とは「スライド全体の面積に対する、テキスト・図形・表が存在しない領域の割合」を指す。padding・行間・コンポーネント間のgapを含む。

##### はみ出し防止ルール（情報密度ガード）

**スライドからコンテンツがはみ出すことは絶対に許容しない。** 以下の上限を守る：

| 要素 | 上限 | 超過時の対応 |
|---|---|---|
| 箇条書き（第1階層） | 5項目以内 | スライドを分割する |
| 箇条書き（第2階層） | 3項目以内 | 省略するか別スライドに |
| 1行の文字数 | 全角30文字以内 | 改行するか表現を短縮 |
| 表の行数 | 6行以内（ヘッダー含む） | スライドを分割する |
| 表の列数 | 5列以内 | 列を統合するか分割 |
| コードブロック | 10行以内 | 省略記号（...）で要点のみ |

##### 改行・折り返しルール

| 場面 | ルール |
|---|---|
| 箇条書き | 1項目1行。長い場合は第2階層に分けず、表現を短縮する |
| 表セル | セル内改行は`<br>`で明示。ただし1セル2行まで |
| 図形・カード内 | テキストは短いフレーズ（10語以内）。説明が必要なら外に出す |
| タイトル（h2） | 1行に収める。長い場合は表現を短縮する（省略不可） |

##### CRAP原則の適用

| 原則 | スライドでの適用 |
|---|---|
| **C: Contrast（コントラスト）** | h3は太字（`###` or `**テキスト**`）で本文と区別。注釈は `<small>` で小さく薄色に。重要な数値はKPIパターンで大きく表示 |
| **R: Repetition（反復）** | 同じ種類の情報は同じパターンで表現。比較は常に2カラム、プロセスは常にフロー。フォントサイズを階層間で混ぜない |
| **A: Alignment（整列）** | Marpのgridレイアウト（columns, compare等）を使用。手動の空白やインデントで位置調整しない |
| **P: Proximity（近接）** | 関連する情報はグルーピング（div, 表, カード）。無関係な情報は `---` で別スライドに分離 |

##### 表のデザインルール

| ルール | 詳細 |
|---|---|
| ヘッダー行 | 太字・背景色付き（テーマCSS自動適用） |
| セル内テキスト | 補足サイズ（17px）で自動適用。長文を入れない |
| 数値列 | 右寄せにしない（Marp制約）。代わりに単位を列名に含める |
| 比較記号 | ◎○△✕ を使う場合は凡例をスライド内に置くか、直感的に分かる記号のみ使用 |

##### 図形・カード内テキストのルール

| ルール | 詳細 |
|---|---|
| テキスト量 | カード1枚あたり1〜2行（フレーズ中心） |
| 見出し | `**太字**` で本文サイズ（20px）、説明文は補足サイズ（17px）で自動適用 |
| フローの矢印 | テーマCSSが自動生成。手動で `→` を入れない（before-afterは例外） |

##### レイアウトパーツ使用時の注意

| パーツ | 注意事項 |
|---|---|
| **横フロー（.flow）** | 矢印はCSSが図形の間に自動配置する。`<small>`注釈はフロー図形の下に配置する（関連する図形要素の近くに置くのが視覚的に自然） |
| **ピラミッド（.pyramid）** | 上3段は白文字、4段目以降は濃色文字。5段以上は使わない（はみ出す） |
| **箇条書き** | CSSで左寄せが自動適用される。centered classのスライド（lead, section, recommendation等）内でも箇条書きは左寄せになる |
| **Before/After** | 中央の `→` は手動で記述する（CSS自動生成ではない） |
| **表** | `table-layout: fixed`のため列幅は均等。特定列を広くしたい場合はcolgroup不可、内容で調整する |

##### 矢印・コネクタの配置原則

**矢印は必ず図形と図形の間に配置する。図形に食い込んではならない。**

- 横フロー（.flow）: CSSが自動で図形間の正確な中央に`→`を配置する。手動で矢印を入れない
- Before/After: `→`は中央のdivに手動で記述する（3カラムgridの中央セル）
- その他のカスタムレイアウト: `margin`または`gap`で図形間に十分なスペースを確保し、そのスペースの中央に矢印を配置する

##### Marp CSSの技術制約

MarpitはCSS内の`rem`単位を`calc(var(--marpit-root-font-size, 1rem) * N)`に自動変換する。**負のrem値**（例: `left: -1.25rem`）はこの変換で壊れるため、位置指定に負の値が必要な場合は**px固定値**を使用すること。テーマCSS編集時の注意事項。

##### テキストのコントラスト（視認性）

**背景色が濃い要素の上に置くテキストは白文字にする。これは絶対ルール。**

| 背景色 | 文字色 | 判定基準 | 例 |
|---|---|---|---|
| 濃色（primary-dark, primary） | **白** | 背景の明度が50%未満 | ピラミッド上段、表ヘッダー、leadスライド |
| 中間色（primary-light, 類似） | **白** | 背景の明度が50〜65% | ピラミッド3-4段目、アクセント付きカード |
| 薄色（bg-sub, lighter, 白系） | **濃色（text）** | 背景の明度が65%以上 | カード内テキスト、表セル、通常スライド |

テーマCSSは以下の濃色背景コンポーネントで白文字を自動適用する：
- ピラミッド（.pyramid > div）: 全段で白文字。`**太字**`も白を継承
- 表ヘッダー（th）: 白文字を自動適用
- leadスライド: 全テキストが白系
- issueスライド: タイトル部分が濃色背景

##### コントラスト確認チェック

スライド生成後、以下を自己チェックする：

1. **濃色背景の上に濃色テキストが載っていないか**: ピラミッド・フロー・カード等の図形内テキストを目視確認
2. **`**太字**`が色を上書きしていないか**: Markdownの`**太字**`は`<strong>`に変換され、CSSの`section strong`ルールで濃色に上書きされる場合がある。テーマCSSで`color: inherit`を設定済みのコンポーネント（pyramid, lead, issue, recommendation, kpi）では問題ないが、カスタムHTMLで濃色背景を使う場合は`<strong style="color:inherit">`または`<b>`を使用する
3. **薄色背景に白文字が載っていないか**: 薄い背景に白文字は読めない

#### 見積書の場合

HTMLを直接生成する。estimate-template.htmlのプレースホルダを埋める。

ファイル名: `YYYYMMDD-見積書.html`（slides-資料/export/ に直接出力）

構成:
- 表紙（プロジェクト名・日付・精度レベル）
- サマリカード（合計工数・期間・チーム人数）
- フェーズ別工数テーブル（小計・合計付き）
- 前提条件・不確定要素

#### スケジュールの場合

HTMLを直接生成する。schedule-template.htmlのプレースホルダを埋める。

ファイル名: `YYYYMMDD-スケジュール.html`（slides-資料/export/ に直接出力）

構成:
- 表紙（プロジェクト名・期間）
- マイルストーンカード
- ガントチャート（ラベル固定+日付横スクロール）
- 凡例
- タスク一覧テーブル

#### ディスカッション資料の場合

HTMLを直接生成する。discussion-template.htmlのプレースホルダを埋める。

ファイル名: `YYYYMMDD-テーマ-discussion.html`（slides-資料/export/ に直接出力）

構成:
- 表紙（テーマ・日付・参加者）
- アジェンダ（時間配分付き）
- トピックカード（テーマ毎に）：
  - 背景ボックス（.context-box）
  - 選択肢カード（.options + .option-card、推奨ありなら .recommended）
  - 質問ボックス（.question-box）
  - 決定欄（.decision-box）
- まとめ・次のアクションテーブル

### 8. テーマCSSの配置（スライドのみ）

スライドの場合、出力先フォルダにテーマCSSをコピーする：
- コピー元: `.aide/templates/export/marp-theme.css`
- コピー先: 各フェーズの `slides-資料/aide-corporate.css`

既に存在する場合はコピーしない。

### 9. VS Code Marp プレビュー設定（スライドのみ）

スライド生成後、VS CodeのMarp for VS Code拡張でリアルタイムプレビューしながら手動修正できるようにする。

ワークスペースルートの `.vscode/settings.json` にMarpテーマ設定が必要。設定がなければ追加する：

```json
{
  "markdown.marp.themes": [
    "./[aideプロジェクトへの相対パス]/.aide/templates/export/marp-theme.css"
  ],
  "markdown.marp.enableHtml": true
}
```

**パスの決定方法:** VS Codeで開いているフォルダ（ワークスペースルート）から、aideプロジェクトの `.aide/templates/export/marp-theme.css` への相対パスを指定する。

例：
- ワークスペースルートがaideプロジェクト自体の場合: `"./.aide/templates/export/marp-theme.css"`
- ワークスペースルートが上位フォルダの場合: `"./プロジェクト名/.aide/templates/export/marp-theme.css"`

設定追加後は `Ctrl+Shift+P` → `Developer: Reload Window` でVS Codeを再読み込みする。

**プレビューの使い方:**
1. 生成されたmdファイルをVS Codeで開く
2. 右上のプレビューボタン（または `Ctrl+Shift+V`）でプレビュー表示
3. mdを編集 → 保存するたびにプレビューがリアルタイム更新
4. テーマCSS（aide-corporate）が反映された状態で確認できる

スライド生成後、以下を案内する：

> mdファイルをVS Codeで開くと、Marp for VS Codeでリアルタイムプレビューしながら手動修正が可能です。

### 10. HTML変換の案内（スライドのみ）

スライドはmdで出力されるため、HTML変換コマンドを案内する：

```bash
marp --html --theme-set [フェーズ]/slides-資料/aide-corporate.css \
  -o [フェーズ]/slides-資料/export/YYYYMMDD-テーマ.html \
  [フェーズ]/slides-資料/YYYYMMDD-テーマ.md
```

aideのスライド出力形式は**HTML**を正とする。HTMLはブラウザでそのままプレゼン可能で、テーマCSSが完全に反映される。

PPTX（PowerPoint）への変換は非推奨。marpの`--pptx`はスライドを画像埋め込みにするため編集不可。編集可能なPPTXを得るにはmdの二重管理が必要になり現実的ではない。PowerPointでの配布が必要な場合はHTMLをPDF化して代替する。

見積書・スケジュール・ディスカッション資料はHTMLで直接生成するため変換不要。

または `/aide-pm-export` でまとめて変換可能。
