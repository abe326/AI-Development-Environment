# aide-ops-fix: タスクの修正対応

指定されたタスクの修正コードを作成する。仕様との整合を確認してから実施する。

## 手順

### 1. タスク情報の確認

- 引数からタスクID（T-XXX）を取得する（未指定の場合はユーザーに確認）
- docs/tasks-タスク/T-XXX-*/ 内の対応記録ファイル（YYYYMMDD-*.md）を読む
- 原因が特定されていない場合は /aide-ops-investigate を先に実行するよう案内する

### 2. 修正前の確認（必須）

以下をユーザーに提示し、承認を得る：

- **対象タスク**: T-XXX のタイトルと原因
- **修正対象**: 変更するファイル・関数
- **修正方針**: どのように修正するか
- **影響範囲**: 変更が及ぶ範囲
- **仕様との整合**: 修正が既存仕様に影響するか

**ユーザーの承認を得てから修正に着手すること。**

### 3. サブエージェントによる並列修正

**Sub-A: 修正コード生成**
- 原因に対する修正コードを作成
- テストも追加する

**Sub-B: 仕様準拠チェック**
- 修正が既存の仕様や動作に影響しないか確認
- 仕様変更が必要な場合は報告

**Sub-C: 影響範囲のテスト実行**
- 既存テストを実行し、パスすることを確認
- 追加したテストも実行

### 4. 記録の更新

- 修正内容のサマリをユーザーに報告
- テスト結果を報告

**対応記録ファイル（docs/tasks-タスク/T-XXX-*/YYYYMMDD-*.md）の対応履歴に追記：**
```markdown
### YYYY-MM-DD 修正

**修正内容:**
- [変更ファイル]: [修正内容]

**テスト結果:** [パス/失敗の状況]

**仕様への影響:** [あり/なし。ありの場合は詳細]
```

また、ファイルの基本情報セクションのステータスを「修正中」に更新する。

### 5. 次のステップの案内

- テストがパスした場合 → レビュー・マージ後に /aide-ops-close でクローズ
- 仕様変更が必要な場合 → /aide-pm-brainstorm で検討を案内

## 規模判断

修正の影響範囲が大きい場合（変更ファイル10超、工数3人日超）は、新規開発フローへの切り替えを提案する。

## 注意事項

- 修正は最小限にする（関係ないリファクタリングを含めない）
- セキュリティに関わる修正は特に慎重に
