# aide-review-audit: 監査レビュー

監査証跡・品質ゲート・コンプライアンス台帳・トレーサビリティマトリクスを横断的に監査し、監査レポートを生成する。Enterpriseモード専用。

## 手順

### 1. ガードレールチェック

CLAUDE.md の「プロジェクト規模」を確認する。

> `quick` / `standard` の場合:
>
> このコマンドはEnterpriseモード専用です。
>
> **代替コマンド:**
> - 成果物の整合性チェック → `/aide-review-doc`
> - コードレビュー → `/aide-review-code`

### 2. 監査スコープの確認

ユーザーに監査スコープを確認する（引数で指定されている場合はスキップ）：

- **`gate`**: 特定の品質ゲートの通過チェック（ゲートIDを指定）
- **`compliance`**: コンプライアンス台帳の全項目チェック
- **`trace`**: トレーサビリティマトリクスの完全性チェック
- **`all`**: 全観点の包括的監査（デフォルト）

### 3. 事前読み込み

1. CLAUDE.md を読む（プロジェクト概要、監査対応セクション）
2. docs/05_audit-監査/compliance-台帳/compliance-register.md を読む
3. docs/05_audit-監査/traceability-トレーサビリティ/traceability-matrix.md を読む
4. docs/05_audit-監査/quality-metrics-品質メトリクス/metrics-history.md を読む
5. docs/05_audit-監査/audit-trail-監査証跡/ 配下のファイル一覧を取得する
6. docs/00_pm-管理/issues-課題.md を読む

### 4. サブエージェントによる並列監査

スコープに応じて該当するサブエージェントのみ実行する。`all` の場合は全て並列実行。

#### Sub-A: 品質ゲートチェック（scope: all, gate）

コンプライアンス台帳の「品質ゲート定義」を参照し、各ゲートの通過条件が満たされているか確認する。

| ゲートID | フェーズ | 通過条件 |
|---|---|---|
| QG-1 | 要件定義完了 | 要件書の存在 + レビュー記録 + OPEN課題0件 |
| QG-2 | 設計完了 | 設計書の存在 + レビュー記録 + トレーサビリティ充足 |
| QG-3 | 実装完了 | コードレビュー記録 + テスト合格 + カバレッジ基準 |
| QG-4 | リリース前 | 全ゲート通過 + 監査レポート承認 |

各ゲートの判定: 通過 / 未通過 / 条件不足（確認不能）

#### Sub-B: 監査証跡の完全性チェック（scope: all）

| チェック項目 | 内容 |
|---|---|
| 証跡の存在 | 各フェーズの主要スキル実行に対応する監査証跡が存在するか |
| 承認記録 | 各監査証跡に承認記録が含まれているか |
| タイムスタンプ整合 | 監査証跡のタイムスタンプが成果物の更新日と整合しているか |
| 不足一覧 | 不足している監査証跡を一覧化 |

#### Sub-C: コンプライアンス台帳チェック（scope: all, compliance）

| チェック項目 | 内容 |
|---|---|
| 対応状況確認 | 全要件（C-XXX）の対応状況を確認 |
| 証跡リンク検証 | 「対応済み」の項目: 証跡リンクが有効か確認 |
| 未確認項目 | 「未確認」の項目: 確認に必要なアクションを提示 |
| 新規要件提案 | 新たに追加すべき要件がないか提案 |

#### Sub-D: トレーサビリティチェック（scope: all, trace）

| チェック項目 | 内容 |
|---|---|
| 要件→設計 | 全要件が設計でカバーされているか |
| 設計→実装 | 全設計が実装でカバーされているか |
| 要件→テスト | 全要件にテストケースが存在するか |
| 未カバー項目 | 「未カバー項目」セクションの更新 |
| カバレッジ率 | カバレッジ率の算出（要件→設計、設計→実装、要件→テスト） |

### 5. 監査レポートの生成

以下の形式で監査レポートを生成する。

保存先: `docs/05_audit-監査/reports-レポート/audit-report-YYYYMMDD.md`

```markdown
# 監査レポート

## 基本情報
- 監査日: YYYY-MM-DD
- 監査スコープ: [all / gate / compliance / trace]
- プロジェクト: [プロジェクト名]

## サマリ

| 監査観点 | 判定 | 指摘件数 |
|---|---|---|
| 品質ゲート | [OK / NG / 一部未通過] | [件数] |
| 監査証跡 | [OK / NG / 一部不足] | [件数] |
| コンプライアンス | [OK / NG / 一部未確認] | [件数] |
| トレーサビリティ | [OK / NG / 一部未カバー] | [件数] |

## 品質ゲート状況

| ゲートID | フェーズ | 状態 | 詳細 |
|---|---|---|---|
| QG-1 | 要件定義完了 | [通過/未通過/条件不足] | [詳細] |
| QG-2 | 設計完了 | ... | ... |
| QG-3 | 実装完了 | ... | ... |
| QG-4 | リリース前 | ... | ... |

## トレーサビリティカバレッジ

| 対応関係 | カバー率 | 未カバー件数 |
|---|---|---|
| 要件 → 設計 | XX% | X件 |
| 設計 → 実装 | XX% | X件 |
| 要件 → テスト | XX% | X件 |

## 指摘事項

| # | 観点 | 重要度 | 内容 | 推奨アクション |
|---|---|---|---|---|
| 1 | [品質ゲート/監査証跡/コンプライアンス/トレーサビリティ] | [Critical/Important/Minor] | [指摘内容] | [推奨アクション] |

## 品質メトリクス（最新）

| メトリクス | 値 | 基準 | 判定 |
|---|---|---|---|
| カバレッジ(C0) | XX% | 80% | [OK/NG] |
| 静的解析(Error) | X件 | 0件 | [OK/NG] |
| セキュリティ(Critical) | X件 | 0件 | [OK/NG] |
```

### 6. コンプライアンス台帳の更新

監査結果に基づいて以下を更新する：
- 確認した項目の「最終確認日」を更新
- 新たに検出した問題の「対応状況」を更新
- 品質ゲートの「状態」を更新

### 7. 課題の起票

Critical の指摘がある場合、ユーザーの確認を得て `docs/00_pm-管理/issues-課題.md` に課題として起票する。監査レポートへのリンクを検討履歴に記載する。

### 8. 次ステップの案内

> 監査レポートを生成しました。
>
> **指摘事項への対応:**
> - Critical指摘がある場合 → 優先的に対応してください
> - トレーサビリティ不足 → `/aide-pm-brainstorm` で仕様を補完
> - テストカバレッジ不足 → `/aide-dev-testspec` でテスト仕様を追加
> - コードレビュー不足 → `/aide-review-code` でレビューを実施
>
> **品質ゲート通過確認:**
> - 全ゲート通過後、リリース準備に進めます

## 注意事項

- Enterpriseモード専用。Standard/Quickモードでは利用不可
- 監査レポートは上書きせず、日付別に蓄積する
- 品質ゲートの通過条件はコンプライアンス台帳で定義・カスタマイズ可能
- 監査証跡が不足している場合、該当スキルの再実行を推奨する（監査証跡は自動生成されるため）
- AIの判断に誤りがある可能性があるため、指摘は人間が確認すること
