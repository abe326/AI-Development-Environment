# aide-router: 対話型ルーター

何をすればいいか迷ったとき、状況に応じて最適なaideコマンドを提案する。

## 手順

### 1. 状況のヒアリング

ユーザーに以下を質問する（わかっている情報はスキップ）：

> 今の状況を教えてください：
>
> 1. **何をしたいですか？**（例: 要件を整理したい、コードを書きたい、バグを調査したい、資料を作りたい）
> 2. **プロジェクトの種類は？** dev（新規開発） / ops（運用保守） / どちらも
> 3. **今どのフェーズですか？**（わからなければ「わからない」でOK）

### 2. フェーズの自動判定（ユーザーが「わからない」と答えた場合）

以下のファイルの存在を確認してフェーズを推定する：

| チェック対象 | 存在する場合のフェーズ |
|---|---|
| docs/03_plans/artifacts-成果物/製造計画書*.md | 実装フェーズ |
| docs/02_design/artifacts-成果物/設計書*.md | 設計フェーズ（計画がなければ） |
| docs/01_requirements/artifacts-成果物/要件書*.md | 要件定義フェーズ（設計がなければ） |
| 上記いずれもなし | 初期化前 or 要件定義の開始 |

CLAUDE.md に「現在のフェーズ」が記載されていればそちらを優先する。

### 3. 規模別の判定分岐

CLAUDE.md の「プロジェクト規模」を確認し、規模に応じた提案を行う：

- **quick の場合**: `/aide-pm-prototype` を優先的に提案する。要件が曖昧・ラフな段階では、ブレストより先にプロトタイプ作成を勧める。プロトタイプ作成後は `/aide-pm-reverse-spec` で仕様を逆生成する流れを提案する
- **standard の場合**: 通常の判定ツリーに従う（デフォルト）
- **enterprise の場合**: フェーズ完了時に `/aide-review-audit` を品質ゲートとして提案する。フェーズ移行の条件として監査レビューの実施を案内する

### 4. コマンドの提案

状況に応じて、以下の判定ツリーで1〜3個のコマンドを提案する。

#### 新規開発（dev）の判定ツリー

```
「プロジェクトをこれから始める」
  → /aide-init dev

「打合せのメモがある」
  → /aide-pm-meeting（議事録化 + 課題起票）

「ラフな要件でまず動くものを作りたい」（quick向け）
  → /aide-pm-prototype（プロトタイプ作成）

「プロトタイプはあるが仕様書がない」
  → /aide-pm-reverse-spec（プロトタイプから仕様を逆生成）

「要件を整理したい / 何を作るか決めたい」
  → /aide-pm-brainstorm（要件定義フェーズ）

「要件はあるが設計がまだ」
  → /aide-pm-brainstorm（設計フェーズ）

「設計はあるが実装計画がまだ」
  → /aide-pm-plan（製造計画書作成）

「見積もりを出したい」
  → /aide-pm-estimate

「コードを書きたい / 実装したい」
  → /aide-dev-code

「テストを書きたい / テスト仕様書を作りたい」
  → /aide-dev-testspec

「テストを実行して証跡を残したい」
  → /aide-dev-test

「コードレビューしたい」
  → /aide-review-code

「ドキュメントの整合性を確認したい」
  → /aide-review-doc

「図を描きたい」
  → /aide-pm-diagram

「資料・スライドを作りたい」
  → /aide-pm-slide

「成果物をWord/PDF/HTMLにしたい」
  → /aide-pm-export

「よくわからないが質問・相談がある」
  → /aide-pm-analyze（ファイル更新なしで対話）

「振り返りをしたい」
  → /aide-pm-retro

「依存ライブラリを更新したい / 脆弱性チェック」
  → /aide-dev-migrate

「フェーズ完了時の監査・品質ゲートを通したい」（enterprise向け）
  → /aide-review-audit
```

#### 運用保守（ops）の判定ツリー

```
「新しい問い合わせ・障害報告が来た」
  → /aide-ops-new

「既存タスクを調査したい」
  → /aide-ops-investigate T-XXX

「調査済みタスクを修正したい」
  → /aide-ops-fix T-XXX

「対応が終わったのでクローズしたい」
  → /aide-ops-close T-XXX

「タスクの状況を確認したい」
  → セッション開始時の自動表示を案内（または tasks-タスク.md を直接確認）
```

### 5. 提案の表示

以下の形式でコマンドを提案する：

> **おすすめコマンド:**
>
> | 優先度 | コマンド | 理由 |
> |---|---|---|
> | 1 | `/aide-pm-brainstorm` | 要件が未整理のため、まず要件をブレストで整理 |
> | 2 | `/aide-pm-diagram` | ブレスト後、図で整理すると理解が深まる |
>
> どれを実行しますか？（番号 or コマンド名で指定）

### 6. 選択されたコマンドの実行

ユーザーが選択したコマンドを実行する。番号指定（「1」）やコマンド名指定（「brainstorm」）のどちらにも対応する。

## 注意事項

- 提案は3個以内に絞る（選択肢が多いと迷う）
- 理由を1行で添える（なぜそのコマンドか）
- ユーザーが「全部教えて」と言った場合は、rules.md のコマンド一覧を表示する
