# aide-pm-analyze: 対話型分析

対話しながら理解を深め、不足事項を指摘する。**ファイルの更新は行わない**。
成果物にまとめたくなったら `/aide-pm-brainstorm` を使う。

## 手順

### 1. 分析対象の確認

ユーザーから以下を確認する：
- 何を分析・整理したいか
- フェーズ（要件定義 / 設計 / 開発）
- 既存資料があれば指定してもらう

### 2. 事前読み込み（コンテキスト最小読み込み）

フェーズに応じて、関連する成果物を読み込む：

**要件定義フェーズ:**
- docs/01_requirements/artifacts-成果物/要件書.md（存在すれば）
- docs/01_requirements/artifacts-成果物/要件書-*.md（存在すれば）

**設計フェーズ:**
- docs/01_requirements/artifacts-成果物/要件書.md
- docs/01_requirements/artifacts-成果物/要件書-*.md（対象業務）
- docs/02_design/artifacts-成果物/設計書.md（存在すれば）
- docs/02_design/artifacts-成果物/設計書-*.md（存在すれば）

**開発フェーズ:**
- docs/02_design/artifacts-成果物/設計書.md
- docs/02_design/artifacts-成果物/設計書-*.md（対象機能）
- docs/03_plans/artifacts-成果物/製造計画書.md（存在すれば）
- docs/03_plans/artifacts-成果物/製造計画書-*.md（存在すれば）

ユーザーが指定した既存資料も読み込む。

### 3. フェーズに応じた分析

#### 要件定義フェーズ — 業務ごとに分析

- 顧客要望を業務ドメインで整理する
- 各業務の目的・背景・アクターを掘り下げる
- 業務フローの正常系・異常系を質問で明確化する
- 業務間の関係性を整理する

#### 設計フェーズ — 機能ごとに分析

- 要件から必要な機能を洗い出す
- 各機能のインターフェース・データの流れを分析する
- 技術的な実現方法を議論する
- 機能間の依存関係を整理する

#### 開発フェーズ — 機能ごとに分析

- 設計書の機能から処理の流れを分析する
- 実装上の懸念点・リスクを洗い出す
- 既存コードとの整合性を確認する

### 4. 不足事項の指摘

分析の過程で、後続フェーズに不足する情報を指摘する。
チェック観点は `/aide-pm-brainstorm` の「不足チェック観点」と同じものを参照する。

### 5. 対話の継続

- ユーザーの質問に答える
- 追加の分析が必要であれば深掘りする
- 理解が深まり成果物にまとめたくなったら `/aide-pm-brainstorm` を案内する

## 注意事項

- **ファイルの更新は一切行わない**。対話のみ
- 分析結果をファイルに残したい場合は `/aide-pm-brainstorm` を使うよう案内する
- コマンドは道具。人間がフローを決める
