# aide-dev-code: 仕様駆動コード作成

仕様を正としてコードを作成・修正する。実装前に必ずユーザーの承認を取る。

## 手順

### 1. 事前読み込み（コンテキスト最小読み込み）

1. docs/00_pm-管理/issues-課題.md を読み、対象の課題・タスクを特定する
2. docs/02_design/artifacts-成果物/設計書.md を読む（設計の全体像）
3. **対象機能の設計書（docs/02_design/artifacts-成果物/設計書-{機能名}.md）を読む** ← 実装の正となる仕様
4. **対象機能の製造計画書（docs/03_plans/artifacts-成果物/製造計画書-{機能名}.md）を読む** ← AIへの実装指示書（実装方針・処理フロー・入出力定義・テストケース）
5. 修正の場合: 対象の既存ソースコードを読む

他の履歴ファイルは読まない。

#### Quickモードの場合

CLAUDE.md の「プロジェクト規模」が `quick` の場合、以下を読む:

1. docs/00_pm-管理/issues-課題.md
2. docs/specs-仕様/artifacts-成果物/仕様書.md（仕様の正）
3. docs/specs-仕様/artifacts-成果物/プロトタイプメモ.md（存在すれば）
4. 修正の場合: 対象の既存ソースコードを読む

設計書・製造計画書の参照は不要（仕様書に統合されているため）。

### 2. 実装前の確認（必須 — 承認なしで実装に進まない）

製造計画書の各セクションを活用して、以下の内容をユーザーに提示し、承認を得る：

- **準拠する仕様**: どの設計書・製造計画書に基づくか（← 製造計画書 §1 基本情報）
- **対象ファイル**: 何を作成/修正するか（← 製造計画書 §2 ファイル構成）
- **処理フロー**: 正常系・異常系の概要（← 製造計画書 §3 処理フロー）
- **入出力**: API/関数のシグネチャ（← 製造計画書 §4 入出力定義）
- **依存関係**: 他機能との接続点（← 製造計画書 §5 依存関係）
- **実装方針**: アプローチの概要（← 製造計画書 §6 実装方針）
- **仕様に記載がない点**: 仕様書に記載がなく、判断が必要な点（あれば仕様の補完を提案）

**ユーザーが「OK」「進めて」等の承認を出すまで実装に着手しないこと。**

### 3. サブエージェントによる並列実装

承認後、以下を並列で実行する：

**Sub-A: 仕様に基づく実装コード生成**
- 設計書・製造計画書に記載された要件・振る舞いに準拠してコードを作成
- 製造計画書の処理フローに従って実装
- 段階的生成: スケルトン → ロジック → エッジケース

**Sub-B: 仕様準拠チェック**
- 生成コードと設計書・製造計画書を照合
- 仕様との差異があれば報告

**Sub-C: 既存テストの実行（あれば）**
- テストを実行し、パスすることを確認

### 4. 結果の統合

- 仕様との差異があれば報告
- コミットメッセージに関連する仕様（設計書・製造計画書のファイル名等）を記載

### 5. 仕様同期チェック（Spec-Anchored）

実装完了後、コード変更が仕様に影響を与えていないか確認する。

#### 5.1 差異検出

実装コードと参照した仕様（設計書・製造計画書。Quickモードでは仕様書.md）を照合し、以下の差異を検出する:

| 差異の種類 | 説明 | 例 |
|---|---|---|
| 追加実装 | 仕様にない機能・処理を追加した | 仕様にないバリデーション追加 |
| 仕様変更 | 仕様と異なる振る舞いで実装した | レスポンス形式が仕様と異なる |
| 仕様未実装 | 仕様にあるが実装されていない | エラーケースの一部が未実装 |
| 暗黙の依存追加 | 新しい外部依存やモジュール間依存を追加した | 新しいライブラリの導入 |

#### 5.2 差異がない場合

「仕様との差異はありません」と報告し、手順6へ進む。

#### 5.3 差異がある場合 — 仕様更新の提案

差異を以下の形式でユーザーに提示する:

> **📋 仕様同期チェック結果**
>
> コード変更により、以下の仕様との差異を検出しました:
>
> | # | 差異の種類 | 対象仕様 | 該当箇所 | 内容 |
> |---|---|---|---|---|
> | 1 | 追加実装 | 設計書-認証.md §3.2 | エラーレスポンス | レート制限エラー(429)を追加 |
> | 2 | 仕様変更 | 製造計画書-認証.md §4.1 | レスポンス形式 | tokenフィールドをネスト構造に変更 |
>
> **推奨アクション:**
> 1. **仕様を更新する** → 対象の仕様ファイルの更新案を生成します
> 2. **コードを仕様に合わせて修正する** → 差異箇所のコードを仕様に合わせて修正します
> 3. **差異を認識した上でそのまま進める** → issues-課題.md に課題として記録します

#### 5.4 ユーザーの選択に応じた処理

**「1. 仕様を更新する」を選択した場合:**

対象の仕様ファイルの更新案を提示し、承認後に仕様を更新する。更新内容はブレスト検討履歴にも記録する（`/aide-pm-brainstorm` で記録するのと同等の形式で brainstorms/ に保存）。

**「2. コードを仕様に合わせて修正する」を選択した場合:**

差異箇所のコードを仕様に合わせて修正する。

**「3. そのまま進める」を選択した場合:**

issues-課題.md に以下の形式で課題を起票する:

```
| I-XXX | 仕様差異 | [対象仕様] と実装に差異あり（[差異内容の要約]） | OPEN | Medium | YYYY-MM-DD |
```

### 6. コーディング規約

- CLAUDE.md に記載されたコーディング規約に従う
- 1PR = 1タスクを原則とする
- コミットメッセージは Conventional Commits に従う

### 7. トレーサビリティマトリクスの更新（Standard / Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `standard` または `enterprise` の場合、traceability-matrix.md の対応する行を更新する:
- Standard: docs/00_pm-管理/traceability-matrix.md
- Enterprise: docs/05_audit-監査/traceability-トレーサビリティ/traceability-matrix.md

ファイルが存在しない場合はスキップする（Quickモードでは不要）。

更新内容: コードファイル列を追加/更新。状態を「実装済み」に更新。

変更影響の提示: トレーサビリティマトリクスの「変更影響マトリクス」を参照し、実装による影響が及ぶ可能性のある成果物を提示する。

### 8. 監査証跡の記録（Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `enterprise` の場合、以下を実行する:

1. 本スキルの実行内容を監査証跡として記録する
2. 保存先: docs/05_audit-監査/audit-trail-監査証跡/YYYYMMDD-HHMMSS-aide-dev-code.md

#### 監査証跡フォーマット

```markdown
# 監査証跡: aide-dev-code 実行記録

## 実行情報
| 項目 | 内容 |
|---|---|
| 実行日時 | YYYY-MM-DD HH:MM |
| スキル | aide-dev-code |
| 実行者 | [CLAUDE.mdの担当者] |
| 対象 | [対象ファイル/機能名] |

## 入力（参照した仕様）
- [参照ファイル一覧]

## 実行結果
- [作成/変更されたファイル一覧]
- [品質チェック結果（該当する場合）]

## 承認記録
- ユーザー承認: [Yes/No]
- 承認内容: [承認した内容の要約]
```

`enterprise` でない場合はスキップする。

## 仕様に不備がある場合

仕様が曖昧・不足・矛盾している場合は、**実装を進めずにユーザーに報告**すること。
必要に応じて /aide-pm-brainstorm で検討し、仕様を補完してから実装する。

## 実装完了後の案内

実装が完了したら、以下の次ステップを案内する：

> コードレビューを実施しますか？ → `/aide-review-code`
> スコープを絞る場合: `/aide-review-code --scope spec`（仕様準拠のみ）等

## 注意事項

- 仕様に記載がない機能を独自判断で追加しない
- セキュリティに関わるコードは特に慎重にレビュー
- `.env` や認証情報はコードに含めない
