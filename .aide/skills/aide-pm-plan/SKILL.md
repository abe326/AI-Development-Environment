# aide-pm-plan: 製造計画書（AIへの実装指示書）の作成

設計書と要件書を元に、製造計画書を作成する。
製造計画書は機能ごとに作成し、AIが実装指示として参照できるレベルの粒度とする。

> **製造計画書の位置づけ**: AIがコードを生成するための「実装指示書」であり、プロジェクト管理文書ではない。
> スケジュール・タスク管理・進捗管理は `docs/00_pm-管理/` で別途管理する。

## 手順

### 1. ガードレールチェック + 事前読み込み

#### ガードレール（警告＋確認。中断はしない）

1. docs/01_requirements/artifacts-成果物/要件書.md が存在しない、または内容が薄い場合 →
   > 「要件書がまだ作成されていません。先に要件整理することを推奨しますが、続行しますか？」

2. docs/02_design/artifacts-成果物/設計書.md が存在しない、または内容が薄い場合 →
   > 「設計書がまだ作成されていません。先に設計整理することを推奨しますが、続行しますか？」

#### 事前読み込み（コンテキスト最小読み込み）

- docs/00_pm-管理/issues-課題.md を読む
- docs/01_requirements/artifacts-成果物/要件書.md を読む（要件の全体像）
- docs/02_design/artifacts-成果物/設計書.md を読む（設計の全体像）
- docs/02_design/artifacts-成果物/設計書-*.md を読む（機能別の設計詳細）
- docs/03_plans/artifacts-成果物/製造計画書.md を読む（既存の計画との整合、存在する場合）
- docs/00_pm-管理/schedule-スケジュール.md を読む（既存のスケジュールとの整合確認、存在する場合）

他の履歴ファイルは読まない。

### 2. 不足チェック

設計書の内容に対して、製造計画を作成するのに十分な情報があるかチェックする。
不足がある場合はユーザーに指摘し、対応方針を確認する。

チェック観点（開発フェーズ）:
- 処理フローが導出できるだけの設計情報があるか
- 実装方針（使用ライブラリ・パターン等）が決まっているか
- テスト方針が言及されているか
- 設計書との対応関係が明確か

### 4. 結果の統合

- docs/03_plans/artifacts-成果物/製造計画書.md にメインを出力
- docs/03_plans/artifacts-成果物/製造計画書-{機能名}.md に機能別計画を出力

> **注意**: スケジュール（schedule.md）やタスク管理（issues.md）への出力はaide-pm-planの範囲外。
> これらは `docs/00_pm-管理/` で別途管理する（`/aide-pm-estimate` 等を使用。課題状況はセッション開始時に自動表示される）。

### 5. ユーザーにレビュー依頼

作成した計画の概要を表示し、以下を確認する：
- 機能の分割粒度は適切か
- 処理フロー・実装方針の粒度はAIへの実装指示として十分か
- 抜け漏れはないか

### 6. トレーサビリティマトリクスの更新（Standard / Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `standard` または `enterprise` の場合、traceability-matrix.md の対応する行を更新する:
- Standard: docs/00_pm-管理/traceability-matrix.md
- Enterprise: docs/05_audit-監査/traceability-トレーサビリティ/traceability-matrix.md

ファイルが存在しない場合はスキップする（Quickモードでは不要）。

更新内容: 製造計画書列を追加/更新。状態を「計画済み」に更新。

### 7. 監査証跡の記録（Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `enterprise` の場合、以下を実行する:

1. 本スキルの実行内容を監査証跡として記録する
2. 保存先: docs/05_audit-監査/audit-trail-監査証跡/YYYYMMDD-HHMMSS-aide-pm-plan.md

## 注意事項

- OPEN課題で計画に影響するものがあれば警告する
- 製造計画書はAIへの実装指示書。タスク管理・スケジュール・進捗管理の要素は含めない
- コマンドは道具。人間がフローを決める
