# aide-pm-retro: 振り返り（レトロスペクティブ）

フェーズ完了時またはプロジェクト完了時に振り返りを実施し、学びを構造化して記録する。

## ユースケース

- フェーズ完了時の中間振り返り
- プロジェクト完了時の最終振り返り
- 問題発生後のポストモーテム

## 手順

### 1. 振り返り対象の確認

ユーザーに以下を確認する（引数で指定されている場合はスキップ）：

- **振り返り種別**: `phase`（フェーズ完了） / `project`（プロジェクト完了） / `incident`（問題発生後）
- **対象フェーズ**（phase 指定時）: requirements / design / plans / implementation / testing
- **参加メンバー**（わかる範囲で）

### 2. 事前読み込み

振り返りの材料として以下を読む：

1. `docs/00_pm-管理/issues-課題.md` — 課題の推移（起票・解決の流れ）
2. `docs/00_pm-管理/schedule-スケジュール.md` — 計画と実績の差異
3. 対象フェーズの `brainstorms-検討履歴/index.md` — 検討の経緯
4. 対象フェーズの `reviews-レビュー/` — レビュー指摘の傾向
5. Git ログ（`git log --oneline --since` で対象期間のコミット傾向）

### 3. AIによる事前分析

ユーザーとの対話の前に、AIが以下を分析して提示する：

#### 定量分析

| 指標 | 計画 | 実績 | 差異 |
|---|---|---|---|
| 成果物数 | X | X | — |
| 課題起票数 | — | X | — |
| OPEN→CLOSED率 | — | XX% | — |
| 検討回数（ブレスト数） | — | X | — |
| 手戻り回数 | — | X（推定） | — |

#### 傾向分析

- 課題の傾向（どのカテゴリが多いか）
- レビュー指摘の傾向（仕様不備 / 品質 / セキュリティ）
- 手戻りが発生した箇所とその原因（推定）

### 4. 対話による振り返り

以下のフレームワークで対話を進める：

#### KPT（Keep / Problem / Try）

1. **Keep（続けたいこと）**: うまくいったこと、今後も継続したい取り組み
2. **Problem（問題だったこと）**: 困ったこと、改善が必要なこと
3. **Try（次に試したいこと）**: Problem に対する具体的なアクション

AIは事前分析の結果をもとに、ユーザーが見落としている観点があれば補足する。
ただし**ユーザーの意見を尊重し、AIの分析を押し付けない**。

### 5. 振り返りレポートの出力

出力先: `docs/00_pm-管理/retro-YYYYMMDD-{対象}.md`

```markdown
# 振り返りレポート

## 基本情報
- 実施日: YYYY-MM-DD
- 対象: [フェーズ名 / プロジェクト全体]
- 参加者: [メンバー]

## 定量サマリ

| 指標 | 値 | 備考 |
|---|---|---|
| 成果物完成数 | X/X | — |
| 課題起票数 | X | うちCLOSED: X |
| 手戻り回数 | X | 主な原因: [要約] |

## Keep（続けたいこと）

1. [良かった点と理由]
2. [良かった点と理由]

## Problem（問題だったこと）

1. [問題点と影響]
2. [問題点と影響]

## Try（次に試したいこと）

| # | アクション | 対象 | 担当 | 期限 |
|---|---|---|---|---|
| 1 | [具体的なアクション] | [対象フェーズ/領域] | [担当者] | [期限] |
| 2 | [具体的なアクション] | [対象フェーズ/領域] | [担当者] | [期限] |

## 学んだこと（Lessons Learned）

- [プロジェクトを超えて活かせる学び]
```

### 6. アクションアイテムの課題化

Try のアクションアイテムを `docs/00_pm-管理/issues-課題.md` に課題として起票する（ユーザーの承認後）。

## 注意事項

- 振り返りは**責任追及ではなくプロセス改善**が目的
- AIの分析は参考情報であり、チームの実感を優先する
- アクションアイテムは具体的・実行可能なものに限定する
- 過去の振り返りレポートがあれば、前回の Try が実行されたか確認する
