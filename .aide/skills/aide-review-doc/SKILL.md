# aide-review-doc: ドキュメントレビュー

各フェーズの成果物を横断的に監査し、整合性・網羅性・トレーサビリティの問題を検出する。

## ユースケース

- フェーズ完了時の品質ゲートチェック
- 顧客提出前のセルフレビュー
- 定期的なドキュメント健全性チェック

## 手順

### 1. 監査対象の確認

ユーザーに以下を確認する（引数で指定されている場合はスキップ）：

- **監査スコープ**: `phase`（特定フェーズ） / `all`（全フェーズ横断）
- **対象フェーズ**（phase 指定時）: requirements / design / plans / testing

### 2. 事前読み込み

監査スコープに応じて deliverables-成果物一覧.md と artifacts-成果物/ を読む。
全フェーズ横断の場合は全フェーズの deliverables を読む。

### 3. サブエージェントによる並列監査

#### Sub-A: 成果物存在チェック

deliverables-成果物一覧.md に定義された成果物が実際に存在するか確認する。

| チェック項目 | 内容 |
|---|---|
| ファイル存在 | deliverables に記載のファイルが artifacts/ に存在するか |
| 状態の正確性 | 「作成済み」なのにファイルが空、または「未作成」なのにファイルがある |
| 必須の欠落 | 必須成果物で未作成のものがないか |

#### Sub-B: 整合性チェック（フェーズ間トレーサビリティ）

フェーズ間で成果物の参照関係が整合しているか確認する。

| チェック項目 | 内容 |
|---|---|
| 要件→設計 | 要件書の各要件が設計書でカバーされているか |
| 設計→製造計画 | 設計書の各機能が製造計画書に含まれているか |
| 製造計画→テスト | 製造計画書のテストケースがテスト仕様書に展開されているか |
| テスト→証跡 | テスト仕様書のテストケースに対応する証跡があるか |

#### Sub-C: 内容品質チェック

各成果物の内容品質を確認する。

| チェック項目 | 内容 |
|---|---|
| 空セクション | 見出しだけで中身がないセクション |
| プレースホルダ残存 | `[...]` `{...}` `TODO` `TBD` が残っていないか |
| 日付の整合性 | 作成日・更新日が論理的に正しいか |
| 用語の一貫性 | 同じ概念に異なる用語を使っていないか |

#### Sub-D: 課題・未決事項チェック

プロジェクト全体の未解決事項を確認する。

| チェック項目 | 内容 |
|---|---|
| OPEN課題 | issues-課題.md に OPEN のまま残っている課題 |
| 未カバー仕様 | テスト仕様書で「未カバー」となっている仕様 |
| ブレスト未決 | brainstorms/ で結論が出ていない検討事項 |

### 4. 監査レポートの出力

出力先: `docs/00_pm-管理/docaudit-YYYYMMDD.md`

```markdown
# ドキュメント監査レポート

## 基本情報
- 監査日: YYYY-MM-DD
- 監査スコープ: [全フェーズ / 特定フェーズ]

## サマリ

| カテゴリ | 問題数 | 重大度内訳 |
|---|---|---|
| 成果物存在 | X | 重大: X / 警告: X |
| 整合性 | X | 重大: X / 警告: X |
| 内容品質 | X | 重大: X / 警告: X |
| 未決事項 | X | 重大: X / 警告: X |
| **合計** | **X** | **重大: X / 警告: X** |

## 判定
- ✅ 提出可 / ⚠️ 要修正（警告のみ） / ❌ 提出不可（重大あり）

## 指摘一覧

| # | 重大度 | カテゴリ | 対象 | 指摘内容 | 推奨対応 |
|---|---|---|---|---|---|
| 1 | 重大 | 整合性 | 設計書-認証.md | 要件R-003に対応する設計が未記載 | 設計書に追記 |
| 2 | 警告 | 品質 | 製造計画書.md | §5にTBDが残存 | 内容を確定 |

## フェーズ別詳細

### 要件定義
- 成果物: X/X 作成済み
- [詳細指摘]

### 設計
- 成果物: X/X 作成済み
- [詳細指摘]

### 製造計画
- 成果物: X/X 作成済み
- [詳細指摘]

### テスト
- 成果物: X/X 作成済み
- 証跡対応率: XX%
- [詳細指摘]
```

### 5. 課題の起票

重大な指摘がある場合、`docs/00_pm-管理/issues-課題.md` に課題として起票する。

### 6. トレーサビリティマトリクスのチェック（Standard / Enterprise モードのみ）

traceability-matrix.md が存在する場合、以下の観点でチェックする:
- 全要件が設計でカバーされているか
- 全設計が製造計画でカバーされているか
- 全製造計画がコードでカバーされているか
- 全要件にテストケースが存在するか
- 「未カバー項目」セクションに記載すべき項目がないか

チェック結果を監査レポートに含める。

### 7. 監査証跡の記録（Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `enterprise` の場合、以下を実行する:

1. 本スキルの実行内容を監査証跡として記録する
2. 保存先: docs/05_audit-監査/audit-trail-監査証跡/YYYYMMDD-HHMMSS-aide-review-doc.md

#### 監査証跡フォーマット

```markdown
# 監査証跡: aide-review-doc 実行記録

## 実行情報
| 項目 | 内容 |
|---|---|
| 実行日時 | YYYY-MM-DD HH:MM |
| スキル | aide-review-doc |
| 実行者 | [CLAUDE.mdの担当者] |
| 対象 | [対象ファイル/機能名] |

## 入力（参照した仕様）
- [参照ファイル一覧]

## 実行結果
- [作成/変更されたファイル一覧]
- [品質チェック結果（該当する場合）]

## 承認記録
- ユーザー承認: [Yes/No]
- 承認内容: [承認した内容の要約]
```

`enterprise` でない場合はスキップする。

## 注意事項

- 監査は読み取り専用（成果物を修正しない）
- 重大度は「提出・品質に直接影響」= 重大、「改善推奨」= 警告
- AIの判断に誤りがある可能性があるため、指摘は人間が確認すること
