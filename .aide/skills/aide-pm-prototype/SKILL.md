# aide-pm-prototype: プロトタイプ生成

ラフな要件からプロトタイプコードを素早く生成する。動作確認後に `/aide-pm-reverse-spec` で仕様を逆生成するフローの起点となる。

## 手順

### 1. ラフ要件の確認

ユーザーから以下を確認する（引数で指定されている場合はスキップ）：

- **何を作りたいか**（1-3行の簡潔な説明でOK）
- **対象ユーザー**（誰が使うか）
- **技術的な制約**（あれば。CLAUDE.mdの技術スタックも参照）
- **参考になるもの**（URL、スクリーンショット、既存サービス等があれば）

すべてが揃わなくても進められる。最低限「何を作りたいか」があればよい。

### 2. ガードレールチェック

#### ガードレール（警告＋選択肢提示。中断はしない）

CLAUDE.md の「プロジェクト規模」を確認する。

`standard` / `enterprise` の場合:

> プロトタイプフローは小規模（quick）モード向けです。通常は仕様を先に作成するフローを推奨します。
>
> **1. このままプロトタイプを作成する**
> → プロトタイプ後に `/aide-pm-reverse-spec` で仕様を逆生成できます
>
> **2. 仕様から始める（推奨）**
> → `/aide-pm-brainstorm` で要件定義から進めます

### 3. 事前読み込み（コンテキスト最小読み込み）

1. CLAUDE.md を読む（技術スタック、プロジェクト概要）
2. docs/00_pm-管理/issues-課題.md を読む（関連する課題があれば）
3. 既存のソースコード（src/）がある場合は構造を把握する

### 4. プロトタイプ方針の提示（必須 — 承認なしで実装に進まない）

以下を整理してユーザーに提示し、承認を得る：

- **スコープ**: 実装する機能（何を含み、何を含まないか）
- **技術選定**: 使用するライブラリ・フレームワーク
- **画面構成**（該当する場合）: 画面一覧と遷移の概要
- **データ構造**: 主要なデータモデルの概要
- **省略するもの**: プロトタイプでは省略する機能・品質要件

**ユーザーが承認するまで実装に着手しないこと。**

### 5. サブエージェントによる並列実行

承認後、以下を並列で実行する：

**Sub-A: プロトタイプコード生成**
- 動くことを最優先とした最小限のコード
- 段階的生成: プロジェクト構成 → 主要機能 → 画面/API → 動作確認手順
- エラーハンドリングは最低限（致命的なもののみ）
- 仮実装の箇所には `// TODO: 本実装で対応` コメントを付与
- セキュリティ上の問題がある仮実装は避ける

**Sub-B: プロトタイプメモの作成**

保存先: docs/specs-仕様/artifacts-成果物/プロトタイプメモ.md（quickの場合）
または docs/01_requirements-要件/artifacts-成果物/プロトタイプメモ.md（standard/enterpriseの場合）

以下の内容で作成する:

```markdown
# プロトタイプメモ

## 基本情報
- 作成日: YYYY-MM-DD
- 目的: [ラフ要件のサマリ]
- 技術スタック: [使用した技術]

## 実装した機能
- [機能1]: [概要]
- [機能2]: [概要]

## 技術的な判断
- [判断1]: [選択肢と理由]
- [判断2]: [選択肢と理由]

## 意図的に省略したもの
- [省略1]: [理由]
- [省略2]: [理由]

## TODOリスト（仮実装・未実装）
- [ ] [TODO1]
- [ ] [TODO2]

## 次のステップ
- [ ] ユーザーの動作確認
- [ ] フィードバック反映
- [ ] `/aide-pm-reverse-spec` で仕様書を逆生成
```

### 6. 結果の統合と動作確認の案内

コードの生成結果を統合し、以下を案内する:

> プロトタイプを生成しました。
>
> **生成したファイル:**
> - [ファイル一覧]
>
> **動作確認方法:**
> - [起動コマンドや確認手順]
>
> **次のアクション:**
> - 修正が必要 → そのまま修正内容を指示してください（このセッションで対応します）
> - 仕様書にまとめたい → `/aide-pm-reverse-spec`
> - さらにブレストしたい → `/aide-pm-brainstorm`
> - コードレビュー → `/aide-review-code`

### 7. イテレーション（必要に応じて）

ユーザーから修正フィードバックを受けた場合:
1. フィードバック内容を確認
2. 修正方針を提示（簡潔に）
3. 承認後に修正を実行
4. プロトタイプメモの「TODOリスト」を更新

イテレーションは何度でも繰り返せる。

## 注意事項

- プロトタイプは「動くこと」が最優先。完成度は求めない
- セキュリティ上の問題がある実装（ハードコードされたパスワード等）は仮実装でも避ける
- プロトタイプメモに判断の記録を残すことで、仕様逆生成時の情報源にする
- 大量のコード生成が必要な場合は、段階的に確認を取りながら進める
