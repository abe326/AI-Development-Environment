# aide共通ルール

> **このファイルはaideフレームワークの共通ルールです。編集しないでください。**
> プロジェクト固有の設定は CLAUDE.md に記載してください。

## 基本原則

1. **Human-in-the-Loop**: AIは提案・生成を担い、人間が判断・承認する。承認なしで実装に進まない
2. **仕様駆動開発（SDD）**: 仕様が唯一の正（Single Source of Truth）。コードは仕様の実装物、テストは仕様の検証物
3. **フェーズ駆動**: フェーズ順に成果物を確定してから次に進む。フェーズ内はイテレーティブ
4. **成果物ベース**: 各ステップで具体的な成果物を出力する。口頭の合意だけで進めない
5. **思考履歴の保全**: ブレストや検討の過程は日付付きファイルで残す
6. **コンテキスト最小読み込み**: 履歴ファイルは全件読まない。最新1-2件のみ、必要時だけ遡る
7. **コンテキスト・ナビゲーション**: brainstorms/index.md を経由して過去のブレストに最短で辿り着く

## ドキュメント読み込みルール

### 常に読むファイル（セッション開始時）
- CLAUDE.md / AGENTS.md
- docs/00_pm-管理/issues-課題.md
- brainstorms/index.md（各フェーズフォルダ内）

### フェーズに応じて読むファイル
- 要件定義作業時 → docs/01_requirements/artifacts-成果物/要件書.md
- 設計作業時 → docs/01_requirements/artifacts-成果物/要件書.md + docs/02_design/artifacts-成果物/設計書.md
- 開発作業時 → docs/02_design/artifacts-成果物/設計書.md + docs/03_plans/artifacts-成果物/製造計画書.md
- テスト作業時 → docs/03_plans/artifacts-成果物/製造計画書.md + docs/04_testing/artifacts-成果物/テスト仕様書*.md

### 最新のみ読むファイル
- 各フェーズの meeting-notes-打合せ/ → 最新1-2件のみ
- brainstorms/ → index.mdで特定した該当ブレスト1-2件のみ

### 履歴を遡るケース（明示的に指示があった場合のみ）
- ブレストの決定理由確認 → 該当ブレストファイル1件の結論セクション
- issues.mdの検討履歴確認 → 該当議事録1件
- 類似の過去対応検索 → ファイル名一覧から1-2件

上記以外の履歴ファイルは読まないこと。

## コーディング規約

- 仕様を正としてコードを作成する
- テストは仕様書から導出する
- 1PR = 1タスク
- コミットメッセージは Conventional Commits + 関連する仕様（設計書の章番号等）
- セキュリティに関わるコードは慎重にレビュー
- `.env` や認証情報をコードに含めない

## 実装時のルール（仕様駆動開発）

- 仕様（設計書・製造計画書）を正として実装する
- 実装前に対象・方針・影響範囲の確認を取る
- AI生成コードが仕様に準拠しているか必ずレビュー
- 仕様に記載がない点はユーザーに確認してから実装する

## 課題管理

### 新規開発（dev）
- 課題は docs/00_pm-管理/issues-課題.md で一元管理
- 課題ID: I-XXX（連番）
- ステータス: OPEN → IN_PROGRESS → RESOLVED → CLOSED / WONTFIX

### 運用保守（ops）
- タスクは docs/00_pm-管理/tasks-タスク.md で一元管理
- タスクID: T-XXX（連番）
- ステータス: OPEN → IN_PROGRESS → RESOLVED → CLOSED / WONTFIX
- 各タスクの対応記録は docs/tasks-タスク/T-XXX-件名/ に格納

## 決定事項管理

- 決定事項は brainstorms/ 内のブレストファイルの「結論」セクションで管理
- 過去の決定を横断的に確認したい場合はAIにブレスト結論を一覧化させる

## 図の運用

- `.drawio.svg` 形式を標準とする
- AIが生成 → 人間がdraw.ioで修正、または 人間が作成 → AIが読み取り

## aideコマンド一覧

`/aide-*` コマンド（Claude Code）またはスキル参照（Copilot/Codex）で以下の業務を実行：

### 共通
| コマンド | 目的 |
|---|---|
| `/aide-init` | プロジェクト初期化（`dev` / `ops` / `dev+ops`） |

### PM・管理（aide-pm-*）
| コマンド | 目的 |
|---|---|
| `/aide-pm-meeting` | 議事録作成 + 課題起票 |
| `/aide-pm-analyze` | 対話で理解を深める・不足を指摘する |
| `/aide-pm-brainstorm` | ブレストして成果物を作成・更新する |
| `/aide-pm-diagram` | draw.io互換SVG図を生成 |
| `/aide-pm-issues` | 課題ステータス確認 |
| `/aide-pm-plan` | スケジュール + 製造計画書作成 |
| `/aide-pm-estimate` | 作業見積もり作成 |
| `/aide-pm-slide` | 資料作成（スライド/見積書/スケジュール/ディスカッション資料の4種類を自動判定） |
| `/aide-pm-export` | 成果物（要件書・設計書等）をHTML化して提出用ドキュメントを生成 |
| `/aide-pm-docaudit` | 成果物の整合性・網羅性・トレーサビリティ監査 |
| `/aide-pm-retro` | 振り返り（KPT）の実施・記録 |

### 開発（aide-dev-*）
| コマンド | 目的 |
|---|---|
| `/aide-dev-code` | 設計書・製造計画書に基づくコード作成/修正 |
| `/aide-dev-testspec` | 仕様からテスト仕様書作成（単体/結合を分離出力） |
| `/aide-dev-test` | テスト実行・証跡収集・レポート生成 |
| `/aide-dev-quality` | 品質レポート生成（カバレッジ・静的解析・セキュリティ） |
| `/aide-dev-migrate` | 依存更新・移行支援（CVE脆弱性チェック含む） |
| `/aide-dev-review` | コードレビュー（仕様準拠チェック） |

### 運用保守（aide-ops-*）
| コマンド | 目的 |
|---|---|
| `/aide-ops-list` | OPEN/IN_PROGRESSタスクの一覧表示 |
| `/aide-ops-new` | 新規問い合わせ受付 + タスク起票 |
| `/aide-ops-investigate` | 指定タスクの調査・原因分析 |
| `/aide-ops-fix` | 指定タスクの修正対応 |
| `/aide-ops-close` | 対応記録 + クローズ |
| `/aide-ops-status` | 全タスクの対応状況サマリ |

## 管理系情報の配置ルール

見積・スケジュール・タスク・優先度・担当・ステータスなどの管理系情報は、成果物（要件書・設計書・製造計画書）には含めない。すべて docs/00_pm-管理/ で一元管理する。

設計フェーズ以降の見積変更は「計画修正」として扱う。利用者の明示的な許可なく変更しない。
