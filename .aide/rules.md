# aide共通ルール

> **このファイルはaideフレームワークの共通ルールです。編集しないでください。**
> プロジェクト固有の設定は CLAUDE.md に記載してください。

## 基本原則

1. **Human-in-the-Loop**: AIは提案・生成を担い、人間が判断・承認する。承認なしで実装に進まない
2. **仕様駆動開発（SDD）**: 仕様が唯一の正（Single Source of Truth）。コードは仕様の実装物、テストは仕様の検証物
3. **フェーズ駆動**: フェーズ順に成果物を確定してから次に進む。フェーズ内はイテレーティブ。フェーズの成果物が概ね完成し次フェーズに移行する際は、ドキュメントレビュー（`/aide-review-doc`）の実施を案内する
4. **成果物ベース**: 各ステップで具体的な成果物を出力する。口頭の合意だけで進めない
5. **思考履歴の保全**: ブレストや検討の過程は日付付きファイルで残す
6. **コンテキスト・ナビゲーション**: brainstorms/index.md を起点に、今回のテーマに関連するブレストだけを選択的に読む。全件読まない
7. **インデックス駆動の読み込み**: 履歴ファイルは index.md のテーマ・結論サマリを見て必要なものだけ読む。「最新N件」ではなく「関連するもの」を基準にする

## コンテキスト管理

### 常に読むファイル（セッション開始時）
- CLAUDE.md / AGENTS.md
- docs/00_pm-管理/issues-課題.md
- brainstorms/index.md（各フェーズフォルダ内）

### フェーズに応じて読むファイル
- 要件定義作業時 → docs/01_requirements/artifacts-成果物/要件書.md
- 設計作業時 → docs/01_requirements/artifacts-成果物/要件書.md + docs/02_design/artifacts-成果物/設計書.md
- 開発作業時 → docs/02_design/artifacts-成果物/設計書.md + docs/03_plans/artifacts-成果物/製造計画書.md
- テスト作業時 → docs/03_plans/artifacts-成果物/製造計画書.md + docs/04_testing/artifacts-成果物/テスト仕様書*.md

### インデックス経由で読むファイル
- 各フェーズの meeting-notes-打合せ/ → 最新1-2件のみ
- brainstorms/ → **まず index.md を読み、今回のテーマに関連するブレストを特定してから読む**。件数制限なし（関連するものはすべて読んでよい）

### 履歴を遡るケース
- ブレストの決定理由確認 → index.md で該当ブレストを特定し、結論セクションを確認
- issues.mdの検討履歴確認 → 該当議事録1件
- 類似の過去対応検索 → ファイル名一覧から関連するものを読む

関連性のない履歴ファイルは読まないこと。

### 会話時の自動グラウンディング

スキル実行時以外の通常会話でも、プロジェクトに関連する話題には文書ベースの裏付けを行う。

#### トリガー判定

以下に該当する場合、グラウンディングを実行する。該当しない場合（一般的なプログラミング知識・雑談）はスキップする。

| 会話中のシグナル | 確認先 |
|---|---|
| 機能名・モジュール名・技術名への言及 | 設計書 + brainstorms/index.md |
| 技術選定の議論（フレームワーク、ライブラリ、DB等） | brainstorms/index.md で過去の選定ブレスト確認 |
| 「なぜこうしたんだっけ」「前に決めた〜」等の過去決定への言及 | brainstorms/index.md → 該当ブレスト |
| 新しいアイデア・代替案の提案 | brainstorms/index.md（過去に検討済みか確認） |
| APIの動作・仕様に関する質問 | 設計書の該当セクション |
| 特定の関数・ファイル・パスへの言及 | 実ファイルの存在確認 |
| 判断に迷う場合 | brainstorms/index.md を確認（軽量） |

#### 確認手順（インデックス駆動）

1. 該当フェーズの brainstorms/index.md のテーマ・結論サマリを確認
2. 関連するブレストがあれば結論セクションのみ読む（全文不要）
3. 仕様に関わる話題は該当の成果物（要件書/設計書/製造計画書）の関連セクションを確認
4. コードの存在・動作に関する主張は実ファイルで確認してから回答する

#### 出力ルール

- **通常時**: 文書の参照を自然に会話に織り込む（「設計書 §3.2 によると…」「3/15のブレストで決定した通り…」）。引用の羅列ではなく、会話の流れに溶け込む形にする
- **矛盾検出時**: 回答前に警告ブロックを表示する:
  > **過去の決定との不一致を検出しました**
  > [ブレスト日付・結論の要約]
  > 変更する場合は `/aide-pm-brainstorm` で検討履歴を残すことを推奨します。
- **該当文書なし**: 通常回答する。決定事項が生まれた場合は `/aide-pm-brainstorm` での記録を案内する
- **コード参照**: 存在しない関数・ファイル・APIを記憶だけで回答しない。実ファイルを確認してから回答する

#### 注意事項

- スキル実行時のフル読み込みとは別。会話時は必要最小限のみ読む
- トークン消費を抑えるため、index.md で関連性を判断してから個別ファイルを読む
- プロジェクト文書が未作成（`/aide-init` 未実行等）の場合はスキップする
- ユーザーが「過去の決定は無視して自由に検討したい」と言った場合は、過去決定を参考情報として提示しつつ制約としない

### セッション開始時の状況表示

セッション開始時、業務タイプに応じて以下の状況分析を自動で実行し表示する。

#### ops を含むプロジェクト

`docs/00_pm-管理/tasks-タスク.md` を読み込み、以下を表示する：

1. **タスク一覧**: IN_PROGRESS → OPEN の優先度順にテーブル表示
2. **警告**: 以下に該当するタスクがあれば強調表示する
   - High優先度で3日以上 OPEN のまま
   - 起票から14日以上経過した OPEN タスク
   - IN_PROGRESS が5件以上（並行しすぎ）
3. **対応状況サマリ**: OPEN / IN_PROGRESS / RESOLVED / CLOSED / WONTFIX の件数
4. **滞留分析**: 7日以上OPENまたは14日以上IN_PROGRESSのタスク、ボトルネック検出
5. **アクション提案**: 最優先で対応すべきタスク、優先度見直しが必要なタスク

#### dev を含むプロジェクト

`docs/00_pm-管理/issues-課題.md` を読み込み、以下を表示する：

1. **課題サマリ**: ステータス別件数（OPEN / IN_PROGRESS / RESOLVED / CLOSED / WONTFIX）
2. **警告**: 以下に該当する課題があれば強調表示する
   - High優先度で7日以上停滞している課題
   - IN_PROGRESS が過多の場合
3. **アクション提案**: 各 OPEN / IN_PROGRESS 課題の次アクション

### Office ファイル読み込みルール

Office形式（.docx / .pptx / .xlsx）のファイルを参照する必要がある場合、以下の手順でMarkdownに変換してから読み込む。

| 拡張子 | 変換コマンド | 必要ツール |
|---|---|---|
| .docx | `pandoc "<file>" -t markdown -o "<output>.md"` | pandoc（未導入時: `python3 .aide/templates/export/scripts/docx2md.py "<file>"` で代替） |
| .pptx | `python3 .aide/templates/export/scripts/pptx2md.py "<file>"` | python-pptx |
| .xlsx | `python3 .aide/templates/export/scripts/xlsx2md.py "<file>"` | openpyxl |

**運用ルール:**
- 変換後のMDファイルは元ファイルと同じディレクトリに `<元ファイル名>.md` として出力する
- 変換済みMDが既に存在し、元ファイルより新しい場合は再変換しない
- 画像が含まれる場合は `<元ファイル名>_images/` ディレクトリに抽出する
- 変換結果はあくまで参考用。正式な成果物としてはMDで再作成すること
- 必要パッケージは `.aide/templates/export/scripts/requirements-import.txt` を参照

### トレーサビリティマトリクス（Standard / Enterprise モードのみ）

成果物間の対応関係を `traceability-matrix.md` で管理する。Quickモードでは使用しない。

**配置先:**

- Enterprise: docs/05_audit-監査/traceability-トレーサビリティ/traceability-matrix.md

**管理する対応関係:**

| 要件ID | 要件概要 | 設計書 | 設計箇所 | 製造計画書 | コードファイル | テスト仕様 | テスト結果 | 状態 |

**更新タイミング:**

| スキル | 更新内容 |
|---|---|
| `/aide-pm-brainstorm`（要件フェーズ） | 要件ID・要件概要を追加。状態を「要件のみ」に設定 |
| `/aide-pm-brainstorm`（設計フェーズ） | 設計書・設計箇所を追加。状態を「設計済み」に更新 |
| `/aide-pm-plan` | 製造計画書列を追加。状態を「計画済み」に更新 |
| `/aide-dev-code` | コードファイル列を追加。状態を「実装済み」に更新 |
| `/aide-dev-testspec` | テスト仕様列を追加 |
| `/aide-dev-test` | テスト結果列を更新 |

**変更影響分析:**

成果物を変更する際、トレーサビリティマトリクスの「変更影響マトリクス」を参照し、影響を受ける成果物をユーザーに提示する。

**要件IDの付番:**

要件書に項目を追加する際、`R-XXX`（連番）のIDを自動付与する。既存の要件書にIDがない場合は、`/aide-pm-reverse-spec` 実行時や `/aide-pm-brainstorm` 実行時に採番する。

## 開発ガイドライン

### コーディング規約

- 仕様を正としてコードを作成する
- テストは仕様書から導出する
- 1PR = 1タスク
- コミットメッセージは Conventional Commits + 関連する仕様（設計書の章番号等）
- セキュリティに関わるコードは慎重にレビュー
- `.env` や認証情報をコードに含めない

### 仕様駆動開発（SDD）ルール

- 仕様（設計書・製造計画書）を正として実装する
- 実装前に対象・方針・影響範囲の確認を取る
- AI生成コードが仕様に準拠しているか必ずレビュー
- 仕様に記載がない点はユーザーに確認してから実装する

### Spec-Anchored（仕様双方向同期）

仕様駆動開発の発展形として、実装と仕様の双方向同期を行う。

- **仕様→実装**: 仕様を正として実装する（従来通り）
- **実装→仕様**: 実装中に仕様外の判断を行った場合、仕様への反映を提案する

`/aide-dev-code` 実行時に仕様同期チェックが自動で行われ、差異が検出された場合はユーザーに以下の選択肢を提示する:
1. 仕様を更新する
2. コードを仕様に合わせて修正する
3. 差異を課題として記録し、そのまま進める

- タスクは docs/00_pm-管理/tasks-タスク.md で一元管理
- タスクID: T-XXX（連番）
- ステータス: OPEN → IN_PROGRESS → RESOLVED → CLOSED / WONTFIX
- 各タスクの対応記録は docs/tasks-タスク/T-XXX-件名/ に格納

### 決定事項管理

- 決定事項は brainstorms/ 内のブレストファイルの「結論」セクションで管理
- 過去の決定を横断的に確認したい場合はAIにブレスト結論を一覧化させる

### 監査・コンプライアンス（Enterprise モードのみ）

CLAUDE.md の「プロジェクト規模」が `enterprise` の場合、以下のルールが適用される。

- **監査証跡**: 各スキル実行時に docs/05_audit-監査/audit-trail-監査証跡/ に実行記録を自動生成する
- **監査証跡の内容**: 実行日時、スキル名、対象、参照した仕様、実行結果、ユーザー承認記録
- **コンプライアンス台帳**: docs/05_audit-監査/compliance-台帳/compliance-register.md で品質要件と品質ゲートを管理する
- **品質ゲート**: 各フェーズ完了時に `/aide-review-audit` で品質ゲートの通過を確認する
- **品質メトリクス**: `/aide-review-code` 実行時にカバレッジ・静的解析・セキュリティのメトリクスを docs/05_audit-監査/quality-metrics-品質メトリクス/metrics-history.md に追記する

## スキル定義の保守

スキル定義は `.aide/skills/<skill-name>/SKILL.md` が **Single Source of Truth**。各AIツール向けのラッパー（`.claude/skills/`、`.agents/skills/`、`.github/skills/`、`.github/prompts/`）は `.aide/scripts/sync-skills.sh` で自動生成する。

**運用ルール:**

- スキル定義を新規作成・編集する場合は、正本 `.aide/skills/<skill-name>/SKILL.md` のみを編集する
- 各ツール向けラッパーは直接編集しない（上書きされる）
- スキル編集後は以下を実行して全ラッパーを再同期する:

```bash
bash .aide/scripts/sync-skills.sh            # 全ツール向けに生成
bash .aide/scripts/sync-skills.sh claude     # Claude Code のみ
bash .aide/scripts/sync-skills.sh copilot    # GitHub Copilot のみ
bash .aide/scripts/sync-skills.sh codex      # Codex CLI のみ
```

**フロントマターの規約:**

| フィールド | 用途 | 必須 |
|---|---|---|
| `name` | スキル識別子（例: `aide-pm-brainstorm`） | ✓ |
| `description` | 長文トリガー（`.aide/skills/` のみ）。発火条件を自然言語で詳細に記述 | ✓ |
| `short_description` | 短文ラベル（ラッパーが参照）。1〜2文、UIに表示される | 推奨 |

`short_description` が無い場合、`sync-skills.sh` はラッパーの `description` に正本の `description` をそのまま流用する。

## 管理系情報の配置ルール

見積・スケジュール・タスク・優先度・担当・ステータスなどの管理系情報は、成果物（要件書・設計書・製造計画書）には含めない。すべて docs/00_pm-管理/ で一元管理する。

設計フェーズ以降の見積変更は「計画修正」として扱う。利用者の明示的な許可なく変更しない。
